Arquitecturas en detalle

Monolítica
Monolith

Todo el código vive en una sola unidad deployable. La forma más simple y directa de arrancar un proyecto. No es necesariamente mala — es la arquitectura correcta para proyectos de tamaño pequeño/mediano.

✓ Pros

  • Fácil de desarrollar y debuggear
  • Sin latencia entre componentes
  • Deploy simple
  • Transacciones ACID simples

✗ Contras

  • Escalabilidad limitada
  • Un cambio requiere redeploy total
  • Puede crecer hasta ser inmanejable
  • Tech stack único
Usar cuando: equipo pequeño, proyecto nuevo, necesitás moverse rápido. Los bots RPA del equipo BPS son esencialmente monolitos — y está bien.
Arquitectura en Capas (Layered)
N-Layer

El código se organiza en capas con responsabilidades definidas: presentación, lógica de negocio, acceso a datos. Cada capa solo habla con la que tiene debajo.

app/
├── routers/       ← Capa de presentación (FastAPI endpoints)
├── services/      ← Capa de lógica de negocio
├── repositories/  ← Capa de acceso a datos
├── models/        ← Modelos de base de datos (SQLAlchemy)
└── schemas/       ← DTOs / validación (Pydantic)
Usar cuando: API REST de tamaño mediano. Es la estructura recomendada para los proyectos FastAPI del equipo.
Hexagonal (Ports & Adapters)
Hexagonal

El dominio de negocio está en el centro, completamente aislado. Se comunica con el exterior a través de puertos (interfaces) y adaptadores (implementaciones). Permite testear el negocio sin DB, sin framework.

app/
├── domain/              ← Entidades y lógica de negocio pura
│   ├── entities.py
│   └── use_cases.py
├── ports/               ← Interfaces (contratos)
│   ├── repositories.py  ← Puerto: "necesito guardar datos"
│   └── notifiers.py     ← Puerto: "necesito notificar"
└── adapters/            ← Implementaciones concretas
    ├── postgres_repo.py ← Adaptador: PostgreSQL
    ├── email_notifier.py
    └── fastapi_router.py
Usar cuando: lógica de negocio compleja que cambia frecuentemente, o cuando necesitás testear sin infraestructura.
Microservicios
Microservices

Cada capacidad de negocio es un servicio independiente con su propia base de datos. Se comunican via HTTP/REST o mensajes asíncronos. Alta complejidad operacional.

# Cada servicio es un repositorio/proyecto independiente
bps-auth-service/        ← Autenticación y usuarios
bps-ratio-service/       ← Generación de ratio reports
bps-notification-service/ ← Emails y alertas
bps-api-gateway/         ← Punto de entrada único
Usar cuando: equipos grandes con ownership por dominio, necesidad real de escalar partes del sistema de forma independiente. No es para equipos pequeños — la complejidad operacional es alta.
Event-Driven (Orientada a Eventos)
Event-Driven

Los componentes se comunican publicando y consumiendo eventos. Totalmente desacoplado — el publicador no sabe quién escucha. Muy relevante para automatizaciones RPA y bots.

# Flujo típico en un bot RPA
Bot termina procesamiento
    → publica evento: "archivo_procesado" { id, status, timestamp }
        → Consumer A: guarda resultado en DB
        → Consumer B: envía email al cliente
        → Consumer C: actualiza dashboard

# Implementación simple con FastAPI + Redis
redis.publish("bot_events", json.dumps({
    "type": "ratio_report_completed",
    "cliente_id": 42,
    "status": "ok"
}))
Usar cuando: procesos asíncronos, múltiples consumidores del mismo evento, desacoplamiento total entre partes del sistema. Muy útil en automatizaciones del equipo BPS.
💡 Regla del equipo BPS: Arrancar con Layered (capas) en todos los proyectos nuevos. Si la complejidad crece y los tests se vuelven difíciles, migrar a Hexagonal. Los microservicios son para cuando una sola app no alcanza más — y eso requiere una decisión de equipo consciente.
