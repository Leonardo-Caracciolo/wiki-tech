🍪 Cookiecutter — Templates de proyectos

Cómo crear, usar y mantener templates de proyectos para estandarizar la estructura en el equipo BPS.

¿Qué es Cookiecutter?

Cookiecutter es una herramienta de línea de comandos que genera proyectos a partir de templates. En vez de copiar y pegar carpetas y renombrar archivos a mano, ejecutás un comando y tenés la estructura completa lista en segundos.

1
Instalación
Una sola vez en el entorno de desarrollo.
# Con pip
pip install cookiecutter

# Con pipx (recomendado — instalación global aislada)
pipx install cookiecutter
2
Usar un template existente
Desde GitHub o carpeta local.
# Desde GitHub
cookiecutter https://github.com/tu-org/bps-template-fastapi

# Desde carpeta local
cookiecutter ./templates/bps-python/

# Usando alias después de configurar
cookiecutter bps-fastapi

Estructura de un template BPS

bps-template-fastapi/ ├── cookiecutter.json ← variables del template ├── {{cookiecutter.project_slug}}/ │ ├── README.md │ ├── pyproject.toml │ ├── .gitignore │ ├── .env.example │ ├── app/ │ │ ├── main.py │ │ ├── config.py │ │ ├── database.py │ │ ├── routers/ │ │ │ └── __init__.py │ │ ├── models/ │ │ │ └── __init__.py │ │ ├── services/ │ │ │ └── __init__.py │ │ └── schemas/ │ │ └── __init__.py │ ├── tests/ │ │ ├── conftest.py │ │ └── test_main.py │ └── docs/ │ └── index.md └── README.md ← doc del template en sí

cookiecutter.json — Variables del template

Este archivo define las preguntas que se le hacen al usuario al generar el proyecto.

{
    "project_name":        "Mi Proyecto BPS",
    "project_slug":        "{{ cookiecutter.project_name.lower().replace(' ', '-') }}",
    "description":         "Descripción del proyecto",
    "author":              "Dev Team BPS",
    "python_version":      "3.11",
    "include_docker":      ["y", "n"],
    "include_github_actions": ["y", "n"],
    "database":            ["postgresql", "sqlite", "none"]
}

Usar variables dentro de los archivos template

# En cualquier archivo del template (ejemplo: README.md)
# {{cookiecutter.project_name}}

{{cookiecutter.description}}

## Instalación

```bash
cd {{cookiecutter.project_slug}}
python -m venv .venv
pip install -r requirements.txt
```

Autor: {{cookiecutter.author}}
# En nombres de archivos y carpetas también
{{cookiecutter.project_slug}}/    ← se convierte en "mi-proyecto-bps/"
{{cookiecutter.project_slug}}.py  ← se convierte en "mi-proyecto-bps.py"

Lógica condicional en templates

{# En cualquier archivo — sintaxis Jinja2 #}
{% if cookiecutter.include_docker == 'y' %}
FROM python:{{cookiecutter.python_version}}-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0"]
{% endif %}

Templates recomendados para el equipo

🐍 bps-fastapi — API REST con FastAPI
FastAPI + PostgreSQL + SQLAlchemy async + pytest. Estructura con routers, services, schemas y models separados. Ideal para nuevas APIs del equipo.
🤖 bps-rpa-bot — Bot de automatización
Estructura para bots RPA con Python. Incluye logging configurado, manejo de excepciones, .env para credenciales y estructura de reportes.
📊 bps-report — Generador de reportes
Template para scripts de generación de reportes Excel/Word. Incluye carpetas input/output/templates y configuración de logging.
💡 Buena práctica: Versioná los templates en un repositorio separado bps-templates y taggueá cada versión. Así los proyectos existentes no se ven afectados por cambios en el template.
