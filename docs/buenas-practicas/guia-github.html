🐙 GitHub — Flujo de trabajo BPS

Convenciones de branching, commits, pull requests y flujo de trabajo del equipo de desarrollo.

Estrategia de branching

El equipo usa GitHub Flow — simple, directo y apropiado para equipos que hacen deploys frecuentes.

main → feature/nombre → PR + review → merge a main → deploy
Principal
main
Rama de producción. Siempre estable. Nunca se pushea directo — todo entra por PR.
Desarrollo
feature/nombre-descriptivo
Una rama por feature o tarea. Se crea desde main y se elimina después del merge.
Urgente
hotfix/descripcion
Para bugs críticos en producción. Se crea desde main, se mergea a main directamente.
Release
release/v1.2.0
Para versiones formales. Permite hacer ajustes finales antes del tag.

Commits — Conventional Commits

Seguimos el estándar Conventional Commits. Permite generar changelogs automáticos y entender el historial de un vistazo.

tipo(scope opcional): descripción corta en minúsculas

# Ejemplos reales del equipo:
feat(ratio-report): agregar soporte para tipo PLI = CB
fix(parser): corregir formato de fechas con locale español
refactor(docx): migrar de pywin32 a python-docx
docs(readme): actualizar instrucciones de instalación
chore(deps): actualizar openpyxl a 3.1.2
test(ratio): agregar tests para filtrado por tipo MB
perf(db): agregar índice en tabla ventas
style: formatear con black
TipoCuándo usarlo
featNueva funcionalidad para el usuario
fixCorrección de bug
refactorCambio de código que no agrega feature ni corrige bug
docsCambios solo en documentación
testAgregar o corregir tests
choreTareas de mantenimiento (deps, config, build)
perfMejora de performance
styleFormato, espacios, punto y coma — sin cambio de lógica
ciCambios en configuración CI/CD

Flujo de trabajo diario

# 1. Siempre partir de main actualizado
git checkout main
git pull origin main

# 2. Crear rama para la tarea
git checkout -b feature/ratio-report-pdf

# 3. Trabajar y commitear en pequeños pasos
git add .
git commit -m "feat(ratio): agregar exportación a PDF"

# 4. Pushear y abrir PR
git push origin feature/ratio-report-pdf
# → Abrir PR en GitHub con descripción clara

# 5. Después del merge, limpiar la rama local
git checkout main
git pull origin main
git branch -d feature/ratio-report-pdf

Pull Requests

Template de PR

## ¿Qué hace este PR?
Descripción clara de los cambios realizados.

## Tipo de cambio
- [ ] Bug fix
- [ ] Nueva feature
- [ ] Refactor
- [ ] Documentación

## ¿Cómo se testea?
Pasos para verificar que funciona correctamente.

## Checklist
- [ ] El código sigue las convenciones del equipo
- [ ] Se agregaron tests si aplica
- [ ] La documentación fue actualizada
- [ ] No hay console.logs o prints de debug
💡 Regla de oro: Un PR debería poder revisarse en menos de 30 minutos. Si es más grande, dividilo en PRs más pequeños.

Comandos útiles

# Ver estado del repositorio
git status
git log --oneline --graph --all

# Deshacer el último commit (sin perder cambios)
git reset --soft HEAD~1

# Traer cambios de main sin perder tu trabajo actual
git stash
git pull origin main
git stash pop

# Rebase interactivo para limpiar commits antes del PR
git rebase -i HEAD~3

# Ver diferencias con main
git diff main...feature/mi-rama

# Buscar en qué commit se introdujo un bug
git bisect start
git bisect bad HEAD
git bisect good v1.0.0
⚠️ Nunca hacer: git push --force en main. En ramas de feature está permitido con --force-with-lease.
