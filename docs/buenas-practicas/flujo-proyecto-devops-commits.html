Flujo E2E — 7 etapas

Principio central: ninguna etapa comienza sin los entregables de la anterior. El flujo es secuencial. Cada salida alimenta la siguiente entrada. No hay atajos.

01 Detección→ 02 Kick-Off→ 03 Relevamiento→ 04 Excepciones→ 05 Sprints→ 06 UAT→ 07 Cierre
01
Detección & Venta
SR Dev
Se identifica la necesidad del cliente o área interna. El SR Dev evalúa viabilidad técnica y fit con las capacidades del equipo. Genera interés en el dueño del problema y obtiene aprobación inicial. Sin esta aprobación formal no existe el proyecto.
Salida obligatoria: Alta de Proyecto firmada por el responsable de negocio
02
Kick-Off
SR + JR Dev
Reunión inicial con todos los involucrados — equipo técnico y responsables del cliente. Se definen objetivos, alcance, participantes clave y fechas tentativas. Todos salen alineados. Si alguien no está alineado, el proyecto fracasa en sprints posteriores.
Salida obligatoria: Documento de alcance con participantes y responsabilidades definidas
03
Relevamiento Detallado
SR Dev + QA
Se documenta el proceso tal como existe hoy (As-Is). Si el proceso no está estandarizado, la automatización no es posible — primero se estandariza el proceso manual, luego se automatiza. QA ingresa al proyecto en esta etapa para entender el proceso desde el inicio. Artefactos: Work Instruction (WI), SOP del área, Diagrama As-Is, User Stories.
Salida obligatoria: Diagrama As-Is aprobado + User Stories validadas
04
Errores & Excepciones
SR + JR Dev + QA
Se mapean todas las excepciones conocidas y se diseñan estrategias para las desconocidas. Se diseña el proceso To-Be con el manejo de errores incluido desde el diseño. Se acuerda con el usuario el comportamiento ante cada tipo de falla.

Tipos de excepción: Business Exception — dato inválido, regla de negocio fallida, caso no contemplado. Application Exception — sistema caído, timeout, error inesperado del entorno.
Salida obligatoria: PDD completo + Diagrama To-Be aprobado por el usuario
05
Sprints Iterativos
Todo el equipo
Ciclos de 1–2 semanas. El MVP del sprint se entrega al usuario al finalizar cada ciclo para validación real. El costo de un cambio en sprint 2 es 10x menor que en producción. El usuario ve resultados reales desde el sprint 1.
Planificación ½ día→ Desarrollo 1-2 sem→ Testing QA→ Demo Usuario→ Retrospectiva
Salida por sprint: MVP funcional con testing completado y aprobado por QA
06
UAT & Validación Final
Todo el equipo
El usuario valida el producto completo en su entorno real con datos reales — no datos de prueba. Se documentan los ajustes finales. QA acompaña todo el proceso de validación. Sin UAT firmado no hay pase a producción.
Salida obligatoria: Documento de UAT firmado por el responsable del área usuaria
07
Cierre & Documentación
Todo el equipo
PDD final actualizado con todos los cambios del proyecto. Documento técnico (SDD). Manual de usuario. Handoff a soporte. Retrospectiva del equipo para registrar aprendizajes que aplican al próximo proyecto.
Salida obligatoria: Paquete completo de documentación entregado y archivado en el repositorio

Reglas de seguimiento del Board

ReglaPor qué importa
📌 Una tarea = un responsableSi dos personas trabajan en algo, se crean dos tareas o una sub-tarea. Nunca responsabilidad compartida sin dueño.
📅 Actualizar el board el mismo díaUn board desactualizado es peor que no tener board. El estado debe reflejar la realidad en tiempo real.
🔴 Bloqueos visibles inmediatamenteSi algo te bloquea, mover a Bloqueado y describir el impedimento ese mismo día. El SR Dev puede desbloquear si lo ve a tiempo.
📝 Decisiones en comentarios de la tareaCambios de alcance, acuerdos con el cliente, decisiones técnicas importantes van como comentario en la tarea. No en Teams, no en el chat.

Definition of Done — Checklist de entrega a QA

Ningún desarrollo puede pasar a "En Revisión / QA" sin completar este checklist. Es el contrato entre DEV y QA.

  • ¿Qué hace el desarrollo? — Descripción en 3 líneas máximo. QA debe entender sin ver el código fuente.
  • ¿Cuáles son los inputs? — Con ejemplos reales. Si procesa Excel, adjuntar un Excel de prueba con datos correctos.
  • ¿Cuáles son los outputs? — Describir exactamente qué se genera: archivo, registro en BD, email, log, etc.
  • ¿Casos borde identificados? — Datos vacíos, formatos incorrectos, valores límite. Documentar cómo se maneja cada uno.
  • ¿Comportamiento ante errores? — Qué mensaje ve el usuario, qué se loguea, si hay rollback o acción compensatoria.
  • ¿Fue probado localmente? — El DEV confirma que funcionó el happy path en su propio entorno antes de entregar.
  • ¿Tiene logs y trazabilidad? — QA debe poder reproducir cualquier problema. Sin logs, no hay debugging posible.
  • ¿Ambiente de QA listo? — Datos de prueba cargados, accesos configurados y dependencias disponibles.
🚫 Si algún ítem del checklist está sin completar, la tarea NO se mueve a "En Revisión / QA". No es una sugerencia. Es el proceso.

Conventional Commits — Estándar del equipo

Un buen historial de commits es documentación gratuita: te dice qué cambió, por qué y quién. Se usa en Python, C# y VBA por igual.

Estructura del mensaje

# LÍNEA 1 — Obligatoria, máximo 72 caracteres
<tipo>(<scope>): <descripción corta en presente, minúsculas>

# CUERPO — Opcional. Por qué se hizo el cambio (no el qué, eso ya está en la línea 1)

# FOOTER — Opcional. Referencia a tarea de Azure DevOps
# Refs: AB#1234

# EJEMPLOS REALES DEL EQUIPO:
feat(ratio-report): agregar soporte para tipo PLI = CB
fix(parser): corregir formato de fechas con locale español
fix(cuit): corregir validación para CUITs con guiones
refactor(docx): migrar de pywin32 a python-docx
docs(readme): actualizar instrucciones de instalación local
chore(deps): actualizar openpyxl a 3.1.2
test(ratio): agregar casos borde para importe negativo y cero
perf(db): agregar índice en columna fecha_emision
style: formatear con black — sin cambios funcionales
ci(azure): agregar pipeline de tests en PR

Versionado semántico

MAJOR.MINOR.PATCH   →   1.4.2

PATCH  → fix: corrección de bug sin romper nada         → 1.4.2 → 1.4.3
MINOR  → feat: nueva funcionalidad compatible            → 1.4.2 → 1.5.0
MAJOR  → breaking change: API incompatible con versión anterior → 1.4.2 → 2.0.0
