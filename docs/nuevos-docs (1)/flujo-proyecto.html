🗂️ Flujo de Proyecto & Buenas Prácticas

Metodología E2E del equipo BPS — desde la detección hasta el cierre. Commits, versionado y checklist de calidad.

Flujo E2E — 7 etapas

Principio clave: ninguna etapa comienza sin los entregables de la anterior. El flujo es secuencial y cada salida alimenta la siguiente entrada.

01 Detección→ 02 Kick-Off→ 03 Relevamiento→ 04 Excepciones→ 05 Sprints→ 06 UAT→ 07 Cierre
01
Detección & Venta
SR Dev
Se identifica la necesidad del cliente o área. Se evalúa viabilidad técnica y fit con las capacidades del equipo. El SR Dev genera interés en el dueño del problema y obtiene la aprobación inicial.
Salida: Alta de Proyecto firmada por el responsable de negocio
02
Kick-Off
SR + JR Dev
Reunión inicial con todos los involucrados: equipo técnico y responsables del cliente. Se definen objetivos, alcance, participantes y fechas tentativas. Todos deben salir alineados.
Salida: Documento de alcance con participantes y responsabilidades definidas
03
Relevamiento Detallado
SR Dev + QA
Se documenta el proceso tal como existe hoy (As-Is). Si el proceso no está estandarizado, no hay automatización posible. QA empieza a participar desde el inicio.
Salida: Diagrama As-Is aprobado + User Stories
04
Errores & Excepciones
SR + JR + QA
Se mapean las excepciones conocidas y desconocidas. Se diseña el proceso To-Be con el manejo de errores incluido. Se acuerda el comportamiento ante cada tipo de falla con el usuario. Tipos: Business Exception (dato inválido) y Application Exception (sistema caído, timeout).
Salida: PDD completo + Diagrama To-Be aprobado
05
Sprints Iterativos
Todo el equipo
Ciclos de 1–2 semanas. Cada sprint incluye planificación, desarrollo, testing y demo al usuario. El MVP se entrega al finalizar cada ciclo para validación real. El costo de un cambio en sprint 2 es 10x menor que en producción.
Planificación ½ día→ Desarrollo 1-2 sem→ Testing QA→ Demo Usuario→ Retro
Salida por sprint: MVP funcional con testing completado
06
UAT & Validación Final
Todo el equipo
El usuario valida el producto completo en su entorno real con datos reales. Se documentan los ajustes finales antes del pase a producción. El QA acompaña todo el proceso.
Salida: Documento de UAT firmado por el responsable del área usuaria
07
Cierre & Documentación
Todo el equipo
PDD final actualizado. Documento técnico (SDD). Manual de usuario. Handoff a soporte. Retrospectiva del equipo para registrar aprendizajes del proyecto.
Salida: Paquete completo de documentación entregado

Commits — Conventional Commits

Seguimos el estándar Conventional Commits. Permite generar changelogs automáticos y entender el historial de un vistazo.

# Formato: tipo(scope): descripción corta en minúsculas

feat(ratio-report): agregar soporte para tipo PLI = CB
fix(parser): corregir formato de fechas con locale español
refactor(docx): migrar de pywin32 a python-docx
docs(readme): actualizar instrucciones de instalación
chore(deps): actualizar openpyxl a 3.1.2
test(ratio): agregar tests para filtrado por tipo MB
perf(db): agregar índice en tabla ventas
style: formatear con black
TipoCuándo usarlo
featNueva funcionalidad para el usuario
fixCorrección de bug
refactorCambio de código sin agregar feature ni corregir bug
docsCambios solo en documentación
testAgregar o corregir tests
choreMantenimiento: deps, config, build
perfMejora de performance
ciCambios en configuración CI/CD

Checklist de cierre de PR

  • El código sigue las convenciones del equipo (naming, estructura)
  • Sin prints de debug ni código comentado innecesario
  • Sin credenciales hardcodeadas — están en .env
  • Tests agregados o actualizados para los cambios
  • Documentación actualizada (README, docstrings, wiki)
  • El PR puede revisarse en menos de 30 minutos
  • Se probó en entorno local con datos reales o de prueba
  • Dependencias actualizadas en requirements.txt / package.json

Definition of Done (DoD)

✅ Código
Revisado por un compañero · Sin deuda técnica obvia · Sigue convenciones · Type hints
✅ Tests
Cobertura ≥ 80% en lógica de negocio · Tests de regresión pasando · Sin tests skippeados sin motivo
✅ Documentación
PDD actualizado · Docstrings completos · README con instrucciones de instalación y uso
✅ QA
UAT completado con el usuario · Casos de excepción probados · Sin issues bloqueantes conocidos
💡 Retrospectiva: Al cierre de cada proyecto o sprint, el equipo documenta qué funcionó bien, qué no y qué va a cambiar en el próximo. Sin retrospectivas, los equipos repiten los mismos errores.
